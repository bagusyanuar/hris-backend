# Dokumentasi API (Swagger & Bruno)

Setiap endpoint API yang dibuat harus didokumentasikan di dua tempat dengan aturan "Split by Domain" (per file untuk setiap domain, bukan monolithic):

**ATURAN WAJIB (STRICT RULES):**
- **Anti-Duplikasi:** AI WAJIB mengecek eksistensi rute/file yang sudah ada (menggunakan fitur *search* atau `ls`) sebelum meng-generate endpoint baru untuk menghindari duplikasi.
- **Konsistensi Variabel URL:** Parameter di URL harus konsisten. Pada dokumentasi Swagger gunakan `{id}`, pada Bruno gunakan `:id`. Jangan gunakan penamaan *custom* yang bisa membingungkan FE (seperti `:employee_id` atau `{{employee_id}}`).
- **Exhaustive Error Responses:** Dokumentasi *Response* tidak boleh asal-asalan. AI WAJIB membaca file `handler.go` untuk mendaftar SEMUA HTTP Error Code yang mungkin terjadi (`200`, `201`, `400`, `404`, `409`, `422`, `500`). Khusus untuk `422 Unprocessable Entity`, wajib menyertakan contoh array error dari validator.
- **API Contract Versioning:** Setiap kali terjadi perubahan, perbaikan, atau penambahan endpoint baru pada API contract, AI **WAJIB** menaikkan versi API contract menggunakan format **Semantic Versioning (SemVer, e.g., MAJOR.MINOR.PATCH)** pada field `info.version` di spesifikasi Swagger YAML serta memperbarui info versi di koleksi Bruno jika dibutuhkan.
- **Nullable vs Always-Present Fields (WAJIB, buat FE/AI agent konsumen API):** Tiap response body sukses (`200`/`201`) WAJIB eksplisit bedakan dua kelas field, supaya konsumen (termasuk AI agent yang scan schema) gak salah asumsi `undefined`/`omit` vs `null`:
  - **Nullable** — field yang bisa `null` (di Go DTO: tipe pointer, mis. `*string`). WAJIB `nullable: true` di schema Swagger + deskripsi singkat kapan null-nya (mis. "null kalau belum diisi").
  - **Selalu ada meski zero-value** — field non-pointer (`string`, `bool`, `int`, slice yang di-inisialisasi `[]T{}` bukan `nil`) WAJIB masuk daftar `required` di schema Swagger, TIDAK boleh nullable. Slice/array response WAJIB ditegaskan "selalu array, gak pernah `null`" di description (selaras `pkg/response` — `data` gak boleh `null` slice).
  - **Cara tentukan:** cek DTO/struct response di `application/dto.go` — field pointer (`*T`) = nullable; field value (`T`) = selalu ada.
  - Definisikan response object sebagai `components.schemas` reusable (`<Entity>Response`, `<Entity>ListResponse`), jangan cuma `example` mentah tanpa `schema` — `example` doang gak bisa dibaca sebagai kontrak tipe/nullability oleh tooling.
  - Di Bruno, tambahkan baris ringkas di `docs { }` sejajar contoh JSON: "Field nullable (bisa `null`): ..." dan "Field yang SELALU ada meski zero-value: ...".
- **Endpoint Purpose Description (WAJIB, buat FE/AI agent yang fetch dokumentasi):** Tiap operation Swagger WAJIB punya `description` (terpisah dari `summary`) + `operationId` unik camelCase (mis. `listCompanies`, `createBranch`):
  - `description` isinya *purpose* (kenapa endpoint ini ada, dipakai buat apa) + *side-effect non-obvious* (mis. auto-demote main branch) + *kapan pakai endpoint ini vs endpoint lain yang mirip*. 2-4 kalimat, JANGAN restate apa yang udah kebaca dari `parameters`/`schema` (mis. gak perlu "page: nomor halaman").
  - `operationId` buat machine-readable identifier — penting kalau dokumentasi ini nanti dikonsumsi tooling agentic (OpenAPI-to-tool/MCP-style), AI gak nebak dari string `summary`.
  - Bruno `docs { }` WAJIB baris pembuka "Tujuan: ..." sejajar `description` Swagger, biar dua dokumen konsisten.
  - **Anti-Drift:** `description` WAJIB direview ulang begitu behavior endpoint berubah (bukan cuma ditulis sekali lalu dilupain) — satu paket sama kewajiban bump versi di poin *API Contract Versioning*.

1. **Swagger OpenAPI (YAML)** di `docs/api/swagger/<domain>.yaml`. Wajib mendeskripsikan `requestBody`, `responses` super komplit, `required` fields, dan `example`. Response sukses WAJIB pakai `components.schemas` (bukan cuma `example`) supaya nullability (lihat poin *Nullable vs Always-Present* di atas) tercatat sebagai kontrak, bukan cuma sample data. Pastikan untuk menghapus rute *dummy/monolith* jika sistem sudah menerapkan pola modular/Progressive Save. AI **WAJIB** mendaftarkan berkas YAML baru ke dalam environment `URLS` pada [docker-compose.yaml](file:///Users/dystopia/go/hris-backend/docs/api/swagger/docker-compose.yaml) agar langsung tampil di Swagger UI lokal.
2. **Bruno Collection** di `docs/api/bruno/<Domain>/<EndpointName>.bru`. Wajib menyertakan blok `docs { ... }` (menggunakan format Markdown) yang menjelaskan endpoint, `required` properties, dan seluruh variasi *Expected Responses* persis seperti yang tercatat di Swagger — termasuk baris nullable vs always-present field (lihat poin *Nullable vs Always-Present* di atas).
